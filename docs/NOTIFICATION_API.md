# BuzzMart Push Notification Infrastructure

BuzzMart implements client push notifications using **Firebase Cloud Messaging (FCM)** via the `firebase-admin` SDK. The system targets customer mobile applications (Android/iOS) and is architected to remain compatible with future customer web applications.

---

## 1. Device Token Lifecycle

```text
Customer Device                 Backend Server                Firebase (FCM)
      │                               │                             │
      │─── 1. Get FCM Token ──────────┼────────────────────────────>│
      │<── 2. Return FCM Token ───────┼─────────────────────────────│
      │                               │                             │
      │─── 3. POST /auth/device ──────>│                             │
      │   (Bind user to token)        │                             │
      │                               │                             │
      │─── 4. FCM Token Refreshed ────┼────────────────────────────>│
      │<── 5. Return New Token ───────┼─────────────────────────────│
      │                               │                             │
      │─── 6. POST /auth/device ──────>│                             │
      │   (Update token on database)  │                             │
```

1. **Obtaining Token**: The mobile application gets a unique registration token using the Firebase Client SDK.
2. **Registration**: When the customer logs in or opens the app, the token is sent to the backend (`POST /api/v1/auth/device`).
3. **Association**: The backend stores the token in the `user_device` table mapped to the authenticated user ID.
4. **Multi-Device Support**: A user can have multiple devices (e.g. tablet + phone). The database holds one row per token.
5. **Token Expiry**: When Firebase reports a token as unregistered (`messaging/registration-token-not-registered`), the `NotificationService` automatically deletes the token row from the database.

## 2. API Endpoints

### Register / Update Device Token

#### Method
```text
POST
```

#### Endpoint
```text
/api/v1/auth/device
```

#### Access
```text
Customer / Authenticated User
```

#### Headers
```http
Authorization: Bearer <access_token>
Content-Type: application/json
```

#### Request Body
```json
{
  "fcmToken": "c1p_Xy8...J2M8"
}
```

#### Response (200 OK)
```json
{
  "success": true,
  "message": "Device registered successfully"
}
```

## 3. Push Payload Format

All push messages generated by [NotificationService.ts](file:///Users/adityaverma/projects/e-commerce/auth_service/src/services/NotificationService.ts) adhere to a clean payload contract. The notification body is visible on the lock screen, while the `data` block contains structured routing information for deep-linking.

### Push Payload Schema

```json
{
  "notification": {
    "title": "Order Shipped!",
    "body": "Your order BZM-20260711-X4D2S has been shipped via BlueDart."
  },
  "data": {
    "notificationId": "uuid-generated-id",
    "type": "ORDER_SHIPPED",
    "entityType": "Order",
    "entityId": "uuid-order-id",
    "route": "/orders/uuid-order-id"
  }
}
```

#### Key Fields:
* **`type`**: The event category whitelisted inside `AuditAction` or matching the matrix.
* **`entityType`**: The target domain (e.g., `Order`, `Product`).
* **`entityId`**: The database identifier of the target entity.
* **`route`**: Navigation URI path inside the mobile/web app.

## 4. Mobile Integration Best Practices

To ensure a seamless user experience, client applications must follow this implementation flow:

1. **Permission Check**: Request permission to show notifications on application startup.
2. **Synchronize Token**: Call `POST /api/v1/auth/device` upon token generation and every time `onTokenRefresh` triggers.
3. **Foreground Listening**: Handle notifications in the foreground using the client-side Firebase listener, displaying customized heads-up overlays.
4. **Tap Handler**: Read the `data` payload of tapped notifications. Use `entityId` and `route` parameters to deep-link directly to the specific screen.
5. **Fresh Fetch**: Do not rely on information passed inside the notification description. Always execute an HTTP GET request to the backend for the current status of the order/product upon screen load.
6. **Session Termination**: Call `DELETE /api/v1/auth/device` or notify the backend of logout to delete the active device token registration and prevent security leaks.
